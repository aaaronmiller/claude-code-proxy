# Feature Specification: OpenRouter Model Scout

**Feature Branch**: `001-openrouter-model-scout`
**Created**: 2026-03-06
**Status**: Draft
**Input**: User description: "OpenRouter Model Scout (ORMS) - A programmatic intelligence-gathering daemon to maintain a localized, enriched database of OpenRouter AI models for intelligent routing decisions based on benchmark scores, cost-efficiency ratios, and performance metrics."

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - View Top Smartest Models (Priority: P1)

**As a** Claude Code Proxy user
**I want to** see the top 5 smartest models available with their scores and costs
**So that** I can select the highest-quality AI for critical reasoning tasks

**Why this priority**: This is the core value proposition - users need to quickly identify the most capable models.

**Independent Test**: Can be verified by running the scout and checking that `leaderboard.json` contains a "smartest" list with at least 5 models, each showing name, intelligence score, and pricing.

**Acceptance Scenarios**:

1. **Given** the scout has run successfully, **When** I view the smartest models leaderboard, **Then** I see at least 5 models ranked by intelligence score descending
2. **Given** a model with a free tier, **When** it appears in the results, **Then** it is clearly marked as free
3. **Given** the current timestamp, **When** I check the data freshness, **Then** all displayed data is no older than 24 hours

---

### User Story 2 - Identify Best Value Models (Priority: P2)

**As a** developer running automated code analysis
**I want to** identify the best price/performance models based on intelligence-to-cost ratio
**So that** I can minimize API costs without sacrificing code quality

**Why this priority**: Cost optimization is important but secondary to finding quality models.

**Independent Test**: Can be verified by checking that the "value models" leaderboard correctly calculates intelligence/cost ratios and excludes free models.

**Acceptance Scenarios**:

1. **Given** a set of models with varying prices, **When** the value score is calculated, **Then** it equals `intelligence_score / prompt_cost` (or another appropriate cost metric)
2. **Given** models with zero cost, **When** the value list is generated, **Then** free models are excluded from value calculation but listed separately
3. **Given** cache pricing information, **When** cost calculations are performed, **Then** cache read/write costs are included in total cost estimates

---

### User Story 3 - Understand Data Gathering Costs (Priority: P3)

**As a** proxy maintainer
**I want to** understand the resource cost of running the data gathering process
**So that** I can budget API usage and maintain operational awareness

**Why this priority**: Operational monitoring is necessary but not blocking for initial release.

**Independent Test**: Can be verified by checking that `meta.json` records token usage, cost estimates, and historical run information.

**Acceptance Scenarios**:

1. **Given** a completed scout run, **When** I examine the run report, **Then** it shows input and output token counts
2. **Given** token usage data, **When** cost calculations are performed, **Then** the estimated cost is provided based on current pricing
3. **Given** multiple runs over time, **When** I review historical data, **Then** past runs are preserved in `meta.json` with timestamps

---

## Edge Cases

- What happens when the OpenRouter API is unavailable or returns errors?
- How does the system handle rate limiting or blocking from web scraping?
- What is the behavior when benchmark data is missing for some models?
- How does the system handle models with pricing changes greater than 10%?
- What happens if the local database is corrupted or missing?
- How does the system handle models that no longer exist in the API response?
- What is the fallback if the leaderboard cannot be generated due to insufficient data?

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST synchronize model data from the OpenRouter API to detect new models, removals, and pricing changes within 30 seconds for approximately 300 models.
- **FR-002**: The system MUST perform conditional deep web scraping on individual model pages to collect benchmark scores (Intelligence, Coding, Agentic) and performance metrics when new models are detected, during weekly audits, or when manually forced.
- **FR-003**: The system MUST implement intelligent change detection using checksums to avoid unnecessary scraping when the model list has not changed.
- **FR-004**: The system MUST flag models with pricing changes exceeding 10% for priority deep re-scanning.
- **FR-005**: The system MUST maintain a local database containing all model metadata including identity fields, context constraints, pricing structures, performance metrics, and benchmark scores.
- **FR-006**: The system MUST generate a `leaderboard.json` file containing at least four top-5 lists: smartest models, best coding models, best free models, and best value models.
- **FR-007**: The system MUST support automatic scheduled execution with configurable thresholds: API sync when last run is older than 24 hours, deep audit when last deep scan is older than 7 days.
- **FR-008**: The system MUST provide a command-line interface with flags: `--force` (full deep scan), `--fast-only` (API only), `--dry-run` (no file writes), `--token-report` (cost details), `--output-format` (json/csv/both), and `--model-filter` (regex selection).
- **FR-009**: The system MUST track token usage and estimated costs for each run in a `meta.json` file with historical preservation.
- **FR-010**: The system MUST handle free models gracefully, excluding them from value ratio calculations to avoid division-by-zero while still listing them in a separate free models category.
- **FR-011**: The system MUST achieve less than 1% block rate during web scraping through the use of anti-detection measures (camoufox-based browser automation).
- **FR-012**: The system MUST complete a full deep scan of 300 models in under 60 minutes.

### Key Entities

- **Model**: AI model available through OpenRouter, identified by unique ID, containing metadata, pricing, performance metrics, and benchmark scores.
- **Leaderboard**: Derived collection of top-5 lists computed from the model database.
- **Meta**: Operational metadata tracking run history, token usage, and cost estimates.
- **Scout**: The main daemon/script responsible for data acquisition, caching, and leaderboard generation.

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Data freshness - all model data in local database is ≤24 hours old during normal operation (measured by timestamp validation).
- **SC-002**: Completeness - ≥95% of active OpenRouter models have complete benchmark data (measured by sampling the model database).
- **SC-003**: Stealth - the scraper achieves <1% block rate on OpenRouter (measured by tracking HTTP responses and retry attempts).
- **SC-004**: Efficiency - a deep scan completes in <60 minutes for a full set of 300 models (measured by wall-clock time from start to finish).
- **SC-005**: Cost transparency - token usage is tracked with ±10% accuracy compared to actual billing (measured by comparing estimated vs actual API costs over multiple runs).
- **SC-006**: Reliability - the system successfully generates all four leaderboard lists with at least 5 entries each (or all available models if fewer than 5 exist with required data).
- **SC-007**: Availability - the API sync completes successfully in <30 seconds even when no changes are detected (measured by performance timing).

---

## Assumptions

- The OpenRouter API at `https://openrouter.ai/api/v1/models` provides stable, documented endpoints for model enumeration.
- Web scraping of individual model pages is permitted for personal/development use within acceptable rate limits.
- The `camoufox` library provides sufficient anti-detection capabilities to maintain <1% block rate.
- The `crawl4ai` parsing library can reliably extract structured data from OpenRouter's HTML pages.
- Local file system storage is sufficient for the model database and leaderboard files.
- Token usage tracking can achieve ±10% accuracy using estimated token counts (no requirement for exact billing reconciliation).
- The system runs in an environment where Python and required dependencies can be installed.
- OpenRouter model pages contain consistent HTML structure that allows reliable data extraction.

---

## Out of Scope

- Real-time model switching or routing logic (this is purely a data gathering tool)
- User interface beyond command-line output and JSON/CSV files
- Integration with the Claude Code Proxy's decision engine (this tool only provides data)
- Distributed deployment or multi-instance coordination
- Advanced analytics beyond the specified leaderboard top-5 lists
- Interactive configuration beyond command-line flags
- Web-based dashboards or visualization tools
- Automatic deployment or containerization
- Support for any AI provider other than OpenRouter
